  - 원칙: 도메인 레이어는 외부 라이브러리 의존 금지, API 키는 서버(BackEnd)에서만 사용

## PART 3. 기술 스택 & 아키텍처 (7~9번 슬라이드)

### 슬라이드 7 — 기술 스택

- **제목**: 사용 기술
- **구성**: 카테고리별 로고 + 한 줄 설명

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| 프론트엔드 | React (Vite) + Tailwind CSS | 빠른 개발, PWA 변환 용이 |
| PWA | vite-plugin-pwa | 앱스토어 없이 홈 화면 설치, 자동 업데이트 |
| 백엔드 | Vercel Serverless Functions | 별도 서버 관리 불필요, 요청 단위 과금 |
| DB/인증 | Supabase (PostgreSQL) | 인증 내장, Row Level Security로 데이터 격리 |
| AI 엔진 | Claude API (Haiku 4.5) | 계획 생성·재조정·코칭 메시지 |
| 알림 | Firebase FCM + Vercel Cron | 정시 자동 푸시 알림 |
| 배포 | Vercel | 프론트+백엔드+Cron 통합, GitHub 연동 자동 배포 |

- **비주얼**: 로고 그리드 (React, Vite, Tailwind, Vercel, Supabase, Claude, Firebase 아이콘)
- **멘트**: "인프라 비용을 최소화하면서도 실서비스 수준으로 동작하는 조합을 골랐습니다. 전부 무료 티어로 운영 가능하고, Claude API 사용량만 소액 과금됩니다."

### 슬라이드 8 — 시스템 아키텍처

- **제목**: 전체 시스템 구조
- **비주얼**: 다이어그램 (아래 구조를 그래픽으로)

```
[사용자 PWA (React)]
        ↕ REST API (JWT 인증)
[Vercel Serverless Functions]
    ├── /api/auth        → 회원가입/로그인
    ├── /api/generate    → Claude API → 계획 생성
    ├── /api/schedule    → 일정 CRUD
    ├── /api/feedback    → Claude API → 코칭 메시지
    └── /api/notify      → FCM 푸시 발송

[Vercel Cron] → 매일 21:00 KST → /api/notify 자동 실행
        ↓
[Firebase FCM] → 사용자 기기 푸시 알림

[Supabase PostgreSQL] ← RLS 적용
    users / goals / schedules / fcm_tokens
```

- **멘트**: "프론트는 백엔드 API만 호출하고, DB나 AI 키 같은 민감 정보는 전부 서버 쪽에서만 다룹니다. 매일 밤 9시엔 크론이 자동으로 돌면서 그날 할 일 안 끝낸 사람에게만 알림을 보냅니다."

### 슬라이드 9 — DB 설계

- **제목**: 데이터베이스 구조
- **비주얼**: 4개 테이블 ERD 간단 도식 (박스 + 관계선)

```
users ──┬── goals ──── schedules (is_review로 학습/복습 구분)
        └── fcm_tokens
```

- **본문**: 각 테이블 핵심 컬럼 1줄씩
  - `goals`: 과목, 시험종류, 마감일, 하루 공부시간
  - `schedules`: 날짜, 제목, 소요시간, 완료여부, 복습회차
  - `fcm_tokens`: 기기별 푸시 토큰
- **보안 포인트**: Row Level Security로 "본인 데이터만" 접근 가능하도록 DB 레벨에서 강제
- **멘트**: "인증뿐 아니라 DB 자체에 보안 정책을 걸어서, 토큰이 유출돼도 다른 사람 데이터에 접근할 수 없게 만들었습니다."

## PART 4. 핵심 기능 소개 F1~F6 (10~13번 슬라이드)

> 4장으로 압축 — F1+F2 한 장, F3+F6 한 장, F4 한 장, F5 한 장

### 슬라이드 10 — F1. 목표 설정 & F2. AI 계획 생성

- **제목**: 목표만 입력하면, AI가 계획을 짭니다
- **좌측(F1)**: 3단계 입력 폼 스크린샷 — 과목/시험종류 → 마감일/하루 공부시간 → 완료 단원/약점
- **우측(F2)**: 생성된 계획 미리보기 스크린샷
- **핵심 포인트**:
  - Claude AI가 **실제 시험 출제 단원**을 기반으로 구체적 태스크 제목 생성 ("1단원 학습" 같은 추상적 제목 아님)
  - 완료한 단원은 재학습 없이 복습만 배치
  - 60일 정보처리기사 계획: 129개 태스크를 약 30초 내 생성 (프로덕션 실측)
- **멘트**: "예를 들어 정보처리기사를 60일 준비한다고 입력하면, AI가 진짜 출제 단원 기준으로 129개의 세부 태스크를 30초 만에 만들어줍니다."

### 슬라이드 11 — F3. 캘린더 & F6. 복습 사이클

- **제목**: 완료하면 끝이 아니라, 복습까지 자동으로
- **좌측(F3)**: 캘린더 화면 스크린샷 — 학습(보라)/복습(주황) 색상 구분
- **우측(F6)**: 복습 사이클 다이어그램

```
학습 완료 ──▶ 1일 후 복습 ──▶ 3일 후 복습 ──▶ 7일 후 복습
        (망각 곡선 기반 자동 삽입)
```

- **핵심 포인트**: 태스크 완료 체크하는 순간, 서버가 자동으로 다음 복습 일정을 캘린더에 삽입
- **멘트**: "공부는 완료 체크로 끝나는 게 아니라, 그 순간부터 복습 스케줄이 자동으로 생성됩니다. 사람이 따로 챙길 필요가 없어요."

### 슬라이드 12 — F4. 푸시 알림

- **제목**: 미루지 못하게, 매일 밤 알림
- **본문**:
  - 매일 21:00 KST, Vercel Cron이 자동 실행
  - 그날 미완료 태스크가 있는 사용자에게만 FCM 푸시 발송
  - "오늘 미완료 학습이 N개 남았어요! 지금 시작해볼까요?"
- **비주얼**: 스마트폰 알림바에 실제 뜬 알림 스크린샷
- **핵심 포인트**: 기기별 토큰 관리, 중복 알림 방지 로직 적용
- **멘트**: "매일 밤 9시에 자동으로 체크해서, 오늘 할 일을 안 끝낸 사람한테만 알림이 갑니다."

### 슬라이드 13 — F5. AI 학습 피드백

- **제목**: 얼마나 했는지, AI가 코칭해줍니다
- **본문**:
  - 주간 달성률, 일별 그래프, 학습 시간 집계
  - 달성률 구간별로 다른 톤의 AI 코칭 메시지 (80%↑ 격려 / 50~80% 조언 / 50%↓ 재조정 제안)
- **비주얼**: 피드백 화면 스크린샷 (원형 그래프 + AI 명언 카드)
- **멘트**: "그냥 숫자만 보여주는 게 아니라, 그 주 성취도에 맞춰서 AI가 다른 톤으로 코칭 메시지를 만들어줍니다."

### 슬라이드 15 — 홈 대시보드

- **비주얼**: 홈 화면 풀샷
- **구성 설명**: 상단 인사말+날짜 → 오늘 태스크 요약 → 달성률 원형 차트 → AI 코칭 카드 → 목표별 태스크 리스트
- **멘트**: "앱 켜자마자 오늘 뭘 해야 하는지, 얼마나 했는지 한 화면에서 다 보입니다."

### 슬라이드 16 — 집중 타이머 & 집중 잠금 모드 (차별화 포인트)

- **제목**: 진짜로 집중하게 만드는 타이머
- **비주얼**: 타이머 화면 스크린샷 (원형 프로그레스 + 잠금 아이콘)
- **핵심 기능 3가지**:
  1. 설정에서 최소 집중 시간 지정 (5/10/15/20/30분 또는 직접 입력)
  2. 최소 시간 채우기 전엔 완료·닫기 버튼 잠금
  3. **앱을 벗어나면 타이머가 즉시 멈춤** — 딴짓하면 시간이 안 감, 돌아오면 "다시 집중하기"로 재개
  4. Wake Lock으로 타이머 중 화면 꺼짐 방지
- **멘트**: "이 부분이 저희 서비스의 차별점입니다. 그냥 타이머가 아니라, 앱을 나가면 시간이 멈추기 때문에 실제로 집중한 시간만 인정됩니다."

### 슬라이드 17 — 캘린더 & 설정

- **비주얼**: 캘린더 화면 + 설정 화면 2컷
- **캘린더**: 월간뷰 완료율 배지, 날짜 클릭 시 상세 팝업
- **설정**: 푸시 알림 on/off, 집중 잠금 시간 설정, 계정 관리
- **멘트**: "캘린더에서 이번 달 전체 진행 상황을 한눈에 보고, 설정에서 본인 학습 스타일에 맞게 조정할 수 있습니다."

## PART 6. 개발 중 마주친 기술적 문제와 해결 (18~20번 슬라이드)

> 발표에서 가장 인상적인 파트가 될 수 있음 — "실제로 디버깅한 과정"을 보여주면 기술력 어필 포인트

### 슬라이드 18 — 문제 해결 사례 ①: 성능 최적화

- **제목**: 로그인이 왜 이렇게 느리지?
- **문제**: 배포 초기 로그인에 1~1.3초 소요
- **원인**: Vercel 함수는 미국(iad1) 리전에서 실행되는데, DB(Supabase)는 서울 리전 → 태평양을 두 번 왕복
- **해결**: `vercel.json`에 `regions: ["icn1"]` 지정 → 서울 리전 고정
- **결과**: 로그인 응답 시간 1.3초 → 0.2~0.5초로 단축
- **비주얼**: Before/After 막대그래프 + 지도 위에 미국↔한국 왕복 경로 화살표
- **멘트**: "서버 로그를 뜯어보니 응답이 태평양을 두 번 건너오고 있었어요. 리전 하나만 바꿨는데 체감 속도가 확 달라졌습니다."

### 슬라이드 19 — 문제 해결 사례 ②: 숨어있던 알림 버그

- **제목**: 왜 알림이 한 번도 안 왔지?
- **문제**: 매일 밤 9시 자동 알림이 크론 등록 이후 한 번도 성공한 적 없었음
- **원인**: Vercel Cron은 실제로는 **GET** 요청을 보내는데, 서버는 **POST만** 허용하고 있어서 매번 405 에러로 거부됨. 수동 테스트는 항상 POST로 했기 때문에 발견이 늦어짐
- **디버깅 과정**: Vercel CLI의 `crons run` 명령으로 실제 크론과 동일한 방식으로 재현 → 로그에서 405 확인
- **해결**: GET·POST 둘 다 허용하도록 수정, 재현 테스트로 200 응답 검증
- **비주얼**: 터미널 로그 캡처 스타일 (405 → 200)
- **멘트**: "겉으로는 다 만들어놓은 것처럼 보였는데, 실제 크론 방식으로 재현해보니 계속 거부되고 있었던 걸 찾아냈습니다. 이런 건 실제 운영 환경에서 검증하지 않으면 절대 못 잡는 버그였어요."

### 슬라이드 20 — 문제 해결 사례 ③: 보안 점검

- **제목**: 최종 점검에서 발견한 보안 구멍
- **문제**: `/api/notify/send`가 인증 없이 누구나 호출 가능한 공개 엔드포인트였음 → 악용 시 전체 사용자에게 스팸 발송 가능
- **해결**: `CRON_SECRET` 환경변수 기반 인증 추가, Vercel Cron은 자동으로 시크릿을 헤더에 실어 보내므로 정상 발송엔 영향 없음
- **그 외 점검 항목**: Supabase RLS 정상 동작(익명 접근 차단), 프론트 번들에 DB 자격증명 미노출, 환경변수 전체 등록 확인
- **멘트**: "기능 다 만들고 끝이 아니라, 배포 직전에 보안 관점에서 한 번 더 점검했고 실제로 구멍을 하나 찾아서 막았습니다."
