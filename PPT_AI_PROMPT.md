**6번 — 개발 방법론**

**7번 — 기술 스택**
표:
| 영역 | 기술 | 선택 이유 |
|---|---|---|
| 프론트엔드 | React 19(Vite 8)+Tailwind CSS 4 | 빠른 개발, PWA 변환 용이 |

| 알림 | Firebase Cloud Messaging+Vercel Cron | 매일 정해진 시각 자동 푸시 |

**8번 — 시스템 아키텍처**
다이어그램으로 표시:
사용자 PWA(React) ↔ REST API(JWT 인증) ↔ Vercel Serverless Functions [`/api/auth` 회원가입·로그인, `/api/generate` Claude API 호출해 계획 생성, `/api/schedule` 일정 CRUD, `/api/feedback` Claude API 호출해 코칭 메시지 생성, `/api/notify/send` FCM 푸시 발송] ↔ Supabase PostgreSQL(RLS 적용, 테이블: users/goals/schedules/fcm_tokens). 별도 화살표로 Vercel Cron(매일 21:00 KST 실행) → `/api/notify/send` 자동 호출 → Firebase Cloud Messaging → 사용자 기기 푸시.

**9번 — DB 설계**
ERD 간단 도식: users 1:N goals, goals 1:N schedules(컬럼 is_review로 학습/복습 구분), users 1:N fcm_tokens. 각 테이블 핵심 컬럼: goals(과목, 시험 종류, 마감일, 하루 공부 가능 시간), schedules(날짜, 제목, 소요 시간, 완료 여부, 복습 회차), fcm_tokens(기기별 푸시 토큰, user_id 1인당 정확히 1개 행 유지). 보안 포인트: RLS로 본인 데이터만 SELECT/UPDATE 가능, 익명 접근 완전 차단.

**18번 — 문제 해결 사례 ① 로그인 응답 속도**
문제: 배포 초기 로그인 요청에 1.0~1.3초 소요. 원인: Vercel 함수 실행 리전(iad1, 미국 버지니아)과 Supabase DB 리전(서울)이 달라 요청이 태평양을 두 번 왕복. 해결: `vercel.json`에 `"regions": ["icn1"]` 지정해 함수 실행 리전을 서울로 고정. 결과: 로그인 응답 시간 1.3초 → 0.2~0.5초로 단축(실측). Before/After 막대그래프로 시각화.

**19번 — 문제 해결 사례 ② 9시 알림이 한 번도 안 온 버그**
문제: 매일 밤 9시 자동 알림을 크론에 등록한 이후 실제로는 한 번도 발송된 적이 없었음(2026-07-02~7/3, 사용자 보고로 발견). 원인: Vercel Cron은 실제로 GET 요청을 보내는데 `/api/notify/send` 핸들러는 POST 요청만 허용하고 있어서 매번 405 Method Not Allowed로 거부됨. 개발 중 수동 테스트는 항상 POST로만 했기 때문에 발견이 늦어짐. 디버깅 과정: `vercel crons run /api/notify/send` 명령으로 실제 Vercel Cron과 동일한 GET 요청을 재현 → `vercel logs`에서 405 응답 확인. 해결: 핸들러가 GET과 POST를 모두 허용하도록 수정 후 재현 테스트로 200 응답과 `{"sent":1,"failed":0}` 반환 확인. 터미널 로그 캡처 스타일로 405→200 전환 시각화.

**20번 — 문제 해결 사례 ③ 최종 점검에서 발견한 보안 구멍**
문제: `/api/notify/send` 엔드포인트가 인증 없이 누구나 호출 가능한 상태였음 → 악의적 요청자가 전체 가입자에게 임의로 푸시 스팸을 발송할 수 있는 취약점. 해결: `CRON_SECRET` 환경변수 기반 Bearer 토큰 인증 추가, Vercel Cron은 자동으로 이 시크릿을 헤더에 담아 보내므로 정상적인 매일 발송 동작에는 영향 없음. 추가 점검 항목: Supabase RLS 정상 동작 확인(익명 접근 시도 시 차단됨), 프론트엔드 빌드 번들에 DB 자격증명·Claude API 키 미노출 확인.

| 비용은 얼마나 드는가? | Vercel/Supabase/Firebase 전부 무료 티어 범위 내에서 운영 중이며, Claude API 호출은 계획 생성·피드백 생성 시에만 발생하고 Haiku 4.5 모델 기준 1회 호출당 비용이 매우 낮음 |
| 왜 네이티브 앱이 아니라 PWA인가? | 앱스토어 심사 절차 없이 즉시 배포·업데이트할 수 있고, 홈 화면 설치형 웹앱으로도 충분한 사용자 경험을 제공할 수 있기 때문 |
| 보안은 어떻게 처리했는가? | JWT 인증, Supabase RLS, API 키는 백엔드 API Routes에서만 사용, Vercel Cron 호출 엔드포인트는 CRON_SECRET Bearer 토큰으로 보호. 최종 점검 단계에서 인증 누락 취약점을 실제로 발견하고 수정한 사례 있음 |

## 발표 전 실행할 테스트 절차 (정확한 명령과 예상 결과)

발표 자료에 넣을 수치(129개 태스크, 30초, 응답시간 등)가 실제로 재현되는지 발표 전날 아래 순서로 직접 검증한다.

### 1. 로그인 응답 속도 확인 (슬라이드 18 근거)

```bash
curl -w "\n총 소요시간: %{time_total}초\n" -o /dev/null -s \
  -X POST https://study-guide-helper.vercel.app/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"<테스트 계정 이메일>","password":"<테스트 계정 비밀번호>"}'
```
기대 결과: 총 소요시간이 0.5초 이하로 나오면 슬라이드 18의 "0.2~0.5초" 수치를 그대로 사용. 1초를 넘으면 실측치로 슬라이드 문구를 수정한다.

### 2. AI 계획 생성 태스크 개수·소요 시간 확인 (슬라이드 10 근거)

PC 크롬으로 시연용 계정 로그인 → 목표 관리 → 새 목표 추가 → 과목 `정보처리기사`, 시험 종류 `자격증`, 마감일 오늘 기준 60일 뒤, 하루 공부 시간 3시간으로 입력 → "AI로 계획 생성" 클릭과 동시에 초 단위로 시간을 잰다(스마트폰 스톱워치 또는 브라우저 개발자 도구 Network 탭의 요청 소요시간 확인). 계획 미리보기 화면에 표시된 총 태스크 개수를 기록한다.
기대 결과: 생성 완료까지 40초 이내, 태스크 개수 100~140개 사이. 이 범위를 벗어나면 슬라이드 10의 "129개, 30초" 수치를 실측값으로 교체한다.

### 3. 푸시 알림 실제 발송 테스트 (슬라이드 12, 19, 스크린샷 07번 근거)

```bash
SECRET=$(grep '^CRON_SECRET' /Users/jiho0215/Study_GuideHelper/BackEnd/.env | cut -d'"' -f2)
curl -s -X GET "https://study-guide-helper.vercel.app/api/notify/send" \
  -H "Authorization: Bearer $SECRET" \
  -w "\nhttp상태: %{http_code}\n"
```
기대 결과: `{"sent":1,"failed":0}`(또는 `sent`가 발송 대상 인원 수)와 `http상태: 200`. 발송 직후 5~10초 이내에 테스트 스마트폰 알림바에 알림이 1개(2개 아님)만 도착하는지 확인 — 도착하면 즉시 07_push_notification.png 캡처. `http상태: 401`이 나오면 `.env`의 `CRON_SECRET`이 Vercel 배포 환경변수와 다르다는 뜻이므로 `vercel env ls`로 재확인한다.

### 4. 복습 태스크 자동 생성 확인 (슬라이드 11 근거)

홈에서 오늘 날짜의 학습 태스크 하나를 "완료" 처리 → 캘린더 탭으로 이동 → 완료한 날짜 기준 1일 후 날짜 칸에 동일 과목의 `[복습]` 태스크가 자동으로 추가돼 있는지 직접 확인한다.
기대 결과: 완료 처리 직후 새로고침 없이(또는 캘린더 재진입 시) 1일 후 날짜에 복습 태스크가 1개 존재. 존재하지 않으면 슬라이드 11 발표 전에 반드시 버그로 재분류하고 수정한다.

### 5. Cron 자동 발송 재현 테스트 (슬라이드 19 사례 재검증용, 선택)

```bash
cd /Users/jiho0215/Study_GuideHelper
vercel crons run /api/notify/send
```
기대 결과: 200 응답. 405가 다시 뜨면 슬라이드 19에서 "해결됨"이라고 발표할 수 없으므로 발표 전 재수정 필요.

### 6. 보안 점검 재확인 (슬라이드 20 근거)

```bash
curl -s -X GET "https://study-guide-helper.vercel.app/api/notify/send" -w "\nhttp상태: %{http_code}\n"
```
(Authorization 헤더 없이 호출)
기대 결과: `http상태: 401`과 `{"error":"인증이 필요합니다"}`. 200이 나오면 인증 로직이 롤백된 것이므로 발표 전 즉시 재수정한다.

### 7. 집중 잠금 모드 동작 확인 (슬라이드 16, 스크린샷 11·12번 근거)

스마트폰에서 설정 → 집중 잠금 10분으로 설정 → 홈에서 태스크 시작 → 타이머 진행 중 홈 버튼을 눌러 앱을 백그라운드로 보냄 → 5초 대기 → 앱으로 복귀. 기대 결과: 타이머가 백그라운드로 보낸 5초만큼 멈춰 있어야 하고(경과 시간에 반영 안 됨), 복귀 시 "다시 집중하기" 또는 이어하기 UI가 보여야 한다. 10분 채우기 전 완료 버튼을 눌러 잠겨서 안 눌리는 것도 함께 확인한다.
